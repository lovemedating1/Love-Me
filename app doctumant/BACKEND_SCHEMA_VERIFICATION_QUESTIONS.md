profiles, sessions, FCM tokens, and presence. Before we write any client
code against it, we need to confirm it matches what's actually deployed —
a few things in it don't match our existing docs/code, and some fields are

## 1. `profiles` table — please send the current full migration SQL

Our existing model only has these columns (from `migration_001.md`): `user_id`,
`name`, `birthday`, `gender`, `orientation`, `interested_in`,
`marital_status`, `relationship_goal`, `hobbies`, `city`, `country`,
`location_lat`, `location_lng`, `location_accuracy_m`,
`distance_preference_km`, `photo_url`, `is_verified`, `ringtone`,
`profile_complete`, `is_premium`, `premium_until`.

Your integration guide's `UserProfile` model includes several columns we've
never seen documented anywhere:

- `photo_verification` (jsonb)
- `is_suspended`, `suspension_reason`, `suspended_at`, `policy_violations`
- `profiles_viewed_count`, `profiles_viewed_reset_at`
- `notify_safety_email`, `notify_safety_push`, `notify_safety_on_triaged`,
  `notify_safety_on_resolved`

**Questions:**
1. Are these columns actually live on the `profiles` table today, or were
   they copy-pasted from a template/spec that hasn't been deployed yet?
2. If they are live: can you send the exact migration file that added them,
   including types, defaults, and nullability?
3. `profiles_viewed_count` / `profiles_viewed_reset_at` — if these are real,
   is there already a `record_profile_view` RPC that increments/resets them,
   or does the client need to read/write these columns directly?
4. The `notify_safety_*` columns — are these meant to replace or extend the
   existing `notification_preferences` table's columns, or are they a
   separate, profile-scoped set of toggles specifically for safety-report
   notifications? Please clarify the intended relationship between the two.
5. `is_suspended`/`suspension_reason`/`policy_violations` — is there an
   admin-side flow that sets these already, and should the client be reading
   `is_suspended` anywhere (e.g. to block login or show a suspended-account
   screen)? This isn't something we've built any UI for yet.

## 2. `active_sessions` table — please send the current migration SQL

Your guide uses columns `session_token`, `device_label`, `user_agent`,
`last_seen_at`. Our own earlier docs (from migration 001) described this
table as supporting `label`, `os`, `last_active`, `is_current` instead.

**Questions:**
1. Which column set is actually live — yours or ours? Please send the exact
   migration SQL so there's no ambiguity.
2. Is `session_token` meant to be generated by the client (as your example
   code does, via a random UUID) or should it be tied to Supabase's own
   session/refresh token? Generating our own random token seems redundant
   with Supabase auth's existing session — what's this field actually for?
3. Is there a unique constraint on `(user_id, session_token)` or similar, and
   what happens on conflict — does inserting a new session for the same
   device create a duplicate row or update in place?
4. You mentioned single-device enforcement is a stated business rule for
   free-tier users — is there already a trigger/RPC enforcing that (e.g.
   revoking older sessions on new login), or does the client need to check
   and revoke manually before creating a new session?

## 3. `fcm_tokens` table — please send the current migration SQL

We requested this table in `BACKEND_PUSH_NOTIFICATIONS_HANDOFF.md` with the
shape `user_id`, `token`, `platform`, `updated_at`. Your integration guide
uses `created_at` instead of `updated_at`, and upserts `onConflict: 'token'`.

**Questions:**
1. Please confirm the exact final column list and which column tracks
   "last updated" (we need this to know when a token was last refreshed —
   `created_at` alone doesn't tell us that if the row is upserted in place).
2. Confirm the actual unique constraint used for the upsert — is it just
   `token` (global uniqueness across all users), or `(user_id, token)`? This
   matters: a global-unique-`token` constraint means if a second install on
   the same device/OS somehow generates the same token, the upsert could
   silently reassign it to a different user's row.
3. Is RLS in place limiting each user to insert/select/update/delete only
   their own rows (`user_id = auth.uid()`)? Please confirm explicitly.
4. Is there already a dispatcher/Edge Function that reads this table and
   actually sends a push via Firebase, or is this table live but nothing
   consumes it yet? (Our push-notifications handoff doc asked for both the
   table and the dispatcher — please confirm which parts are actually done.)

## 4. `user_presence` table — please send the current migration SQL

Your guide's shape: `user_id`, `is_online`, `last_seen`, upserted on
`user_id` conflict.

**Questions:**
1. Please confirm this exact shape is what's live.
2. Is there any server-side staleness sweep (e.g. a cron marking a user
   offline if no heartbeat received in N minutes), or is `is_online` purely
   whatever the client last set — meaning a client that crashes/loses
   connection without calling `setOffline()` would show as online forever?
3. What heartbeat interval do you recommend the client use, if any?
