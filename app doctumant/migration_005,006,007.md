# LoveMe Flutter — Profile Photos Integration Guide

Covers `009_profile_photos.sql` + `010_profile_photos_rls.sql` (table + RLS), the photo-related trigger in `011_triggers_core.sql` / `012_triggers_core_fixes.sql`, and RPC 10 (`set_primary_profile_photo`) from `013_rpc_functions.sql`.

**Verification method for this doc:** static trace through the actual migration files in this repo (read line-by-line, not from memory or spec), cross-checking table definitions, constraints, RLS policies, trigger bodies, and RPC bodies against each other. This was **not** run against a live database — unlike `CHAT_INTEGRATION_GUIDE.md` (verified live on 2026-07-06), no live Supabase project was queried for this doc. Treat the "Correctness trace" section as a reviewed-on-paper result, and smoke-test the upload → set-primary → discover-feed flow against a real project before shipping.

---

## 1. What's new

| Object | File | Purpose |
|---|---|---|
| `profile_photos` (table) | `009_profile_photos.sql` | Gallery of up to 4 photos per user |
| RLS on `profile_photos` | `010_profile_photos_rls.sql` | Open read, owner-only write |
| `set_updated_at()` trigger on `profile_photos` | `011_triggers_core.sql`, renamed in `012` | Maintains `updated_at` |
| `sync_primary_profile_photo()` trigger | `011_triggers_core.sql` | Mirrors the primary photo's URL onto `profiles.photo_url`, unsets any other primary |
| `set_primary_profile_photo(photo_id)` RPC | `013_rpc_functions.sql` | The function Flutter should call to change the primary photo |

`profiles.photo_url` is still the single column the rest of the app (discover feed, matches, chat list) reads for "the" profile photo. `profile_photos` is the gallery; exactly one row per user can have `is_primary = true`, and that row's `photo_url` is what gets copied into `profiles.photo_url`.

## 2. Table schema

```sql
create table public.profile_photos (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  photo_url text not null,
  display_order integer not null,       -- 1 to 4
  is_primary boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

Constraints you must satisfy from the client:

| Rule | Enforced by | Error code |
|---|---|---|
| `photo_url` cannot be empty/whitespace | `profile_photos_photo_url_not_empty` | `23514` |
| `display_order` must be 1–4 | `profile_photos_display_order_range` | `23514` |
| At most one row per `(user_id, display_order)` slot → caps a user at 4 photos | `profile_photos_user_id_display_order_key` (unique) | `23505` |
| At most one primary photo per user, enforced even under concurrent writes | `idx_profile_photos_one_primary_per_user` (unique partial index on `user_id where is_primary = true`) | `23505` |

**Important for upload UI:** because `(user_id, display_order)` is unique, you cannot insert a 5th photo at a used slot — pick the next free `display_order` in `{1,2,3,4}` client-side (query existing rows first), or let the insert fail with `23505` and let the user replace/reorder instead.

## 3. RLS summary

| Action | Who | Policy |
|---|---|---|
| SELECT | anyone (`public`, including anonymous) | `"Open select profile photos"` — photos must be visible on other users' profiles during discovery |
| INSERT | authenticated, own rows only | `"Users can insert own profile photos"` — `auth.uid() = user_id` |
| UPDATE | authenticated, own rows only | `"Users can update own profile photos"` |
| DELETE | authenticated, own rows only | `"Users can delete own profile photos"` |

A user can never read/write another user's row for INSERT/UPDATE/DELETE — PostgREST returns `403` (RLS) or the write silently affects 0 rows depending on how the request is shaped. As with `messages` (see `CHAT_INTEGRATION_GUIDE.md` §8), **check the response row count**, don't assume `200` means the write applied.

## 4. Triggers involved

### 4.1 `profile_photos_set_updated_at` (BEFORE UPDATE)
Generic — just stamps `updated_at = now()`. Function: `public.set_updated_at()`. Nothing client-visible to worry about; you never need to send `updated_at` yourself.

### 4.2 `sync_primary_profile_photo` (AFTER INSERT OR UPDATE, `WHEN (new.is_primary)`)

```sql
-- fires only when the affected row's is_primary is true
if new.is_primary then
  update public.profile_photos
  set is_primary = false
  where user_id = new.user_id and id <> new.id and is_primary = true;

  update public.profiles
  set photo_url = new.photo_url
  where user_id = new.user_id;
end if;
```

What this means for the client:
- Inserting a photo with `is_primary: true` directly (e.g. the very first photo a user uploads) will **automatically** update `profiles.photo_url` — you do not need a separate call.
- The trigger's own internal `UPDATE ... SET is_primary = false` on other rows does **not** recurse, because that statement never sets `is_primary = true`, so the `WHEN (new.is_primary)` guard on that re-fired trigger evaluates false. This is intentional, not an oversight.

## 5. RPC: `set_primary_profile_photo(photo_id uuid)`

This is the function the client should call to **switch** which existing photo is primary (as opposed to setting `is_primary: true` on insert, which also works but is only relevant for the first photo).

```sql
create or replace function public.set_primary_profile_photo(photo_id uuid)
returns void
language plpgsql
security invoker
as $$
declare
  v_owner uuid;
begin
  if auth.uid() is null then
    raise exception 'Not authenticated';
  end if;

  select user_id into v_owner
  from public.profile_photos
  where id = set_primary_profile_photo.photo_id
  for update;

  if not found then
    raise exception 'Photo not found';
  end if;

  if v_owner <> auth.uid() then
    raise exception 'Not the owner of this photo';
  end if;

  update public.profile_photos
  set is_primary = false
  where user_id = auth.uid() and id <> set_primary_profile_photo.photo_id and is_primary = true;

  update public.profile_photos
  set is_primary = true
  where id = set_primary_profile_photo.photo_id;
end;
$$;
```

Behavior:
1. Rejects unauthenticated calls (`Not authenticated`).
2. Rejects a `photo_id` that doesn't exist (`Photo not found`).
3. Rejects a `photo_id` owned by someone else (`Not the owner of this photo`) — this is a defense-in-depth check; RLS would also block a direct table UPDATE on someone else's row, but the RPC checks explicitly so the error message is meaningful instead of a generic RLS `403`.
4. Unsets the previous primary (if any), then sets the new one — which fires trigger 4.2 and updates `profiles.photo_url` automatically.

Call it with Supabase Flutter's `.rpc()`:

```dart
await supabase.rpc('set_primary_profile_photo', params: {'photo_id': photoId});
```

`security invoker` means it runs as the calling user, so normal RLS on `profile_photos` and `profiles` still applies underneath it — the function's own ownership check and RLS are redundant-but-consistent, not conflicting.

## 6. Correctness trace (static — see verification note at top)

Walked the full chain for the two flows that touch this feature:

**Flow A — first photo upload (INSERT with `is_primary: true`)**
1. Insert satisfies `photo_url_not_empty`, `display_order_range`, unique `(user_id, display_order)`, unique-partial `(user_id) where is_primary` (no existing primary row, so no conflict).
2. RLS `INSERT` policy checks `auth.uid() = user_id` — passes if the client sends its own `user_id`.
3. Trigger fires (`new.is_primary = true`): no other primary rows to unset; `profiles.photo_url` is set. Passes RLS on `profiles` UPDATE (`auth.uid() = user_id`) because the trigger runs `SECURITY INVOKER` and the row being updated is the caller's own profile.
4. Result: consistent state, one primary photo, `profiles.photo_url` in sync.

**Flow B — switching primary via RPC**
1. RPC verifies auth + ownership, locks the row (`FOR UPDATE`) to avoid a race with a concurrent `set_primary_profile_photo` call on the same user unsetting/setting different rows simultaneously.
2. Unset-others `UPDATE`: 0 or 1 row affected (the partial unique index guarantees at most one prior primary); trigger does not fire meaningfully (`WHEN` guard false on the newly-false rows).
3. Set-new-primary `UPDATE`: trigger fires, unsets others again (already unset — no-op), updates `profiles.photo_url`.
4. Result: exactly one primary row, `profiles.photo_url` matches it. **No bug found in this path.**

**Edge case flagged, not a bug in current usage:** the trigger's `UPDATE public.profiles` runs as the invoker (whoever's session fired the INSERT/UPDATE on `profile_photos`), not as a definer. Today, every write path to `profile_photos` (RLS INSERT/UPDATE policies, and RPC 10's ownership check) guarantees `new.user_id = auth.uid()`, so the profiles RLS update policy (`auth.uid() = user_id`) always passes. If a future service-role/admin process ever inserts or updates a `profile_photos` row on behalf of another user (bypassing RLS via the service key), the trigger's `profiles` update would still succeed too, because the service role bypasses RLS entirely — so this is not actually exploitable, just worth knowing the trigger has no explicit `SECURITY DEFINER` escalation and doesn't need one given current call sites.

**Things NOT covered by any current object** (gaps, not bugs):
- No trigger/constraint prevents a user from having **zero** primary photos (e.g. after deleting the primary one, no row is auto-promoted). `profiles.photo_url` would then simply keep its last-synced value until the user picks a new primary — handle "no primary photo" gracefully in the UI rather than assuming one always exists.
- No RPC exists yet for uploading a photo (client inserts directly into `profile_photos`) or reordering `display_order` — only `set_primary_profile_photo` exists as an RPC. Direct table INSERT/UPDATE/DELETE under RLS is the intended path for those operations per the current migrations.

## 7. Client usage examples

**models/profile_photo.dart**

```dart
class ProfilePhoto {
  final String id;
  final String userId;
  final String photoUrl;
  final int displayOrder;
  final bool isPrimary;
  final DateTime createdAt;
  final DateTime updatedAt;

  ProfilePhoto({
    required this.id,
    required this.userId,
    required this.photoUrl,
    required this.displayOrder,
    required this.isPrimary,
    required this.createdAt,
    required this.updatedAt,
  });

  factory ProfilePhoto.fromJson(Map<String, dynamic> json) => ProfilePhoto(
        id: json['id'],
        userId: json['user_id'],
        photoUrl: json['photo_url'],
        displayOrder: json['display_order'],
        isPrimary: json['is_primary'],
        createdAt: DateTime.parse(json['created_at']),
        updatedAt: DateTime.parse(json['updated_at']),
      );
}
```

**services/profile_photo_service.dart**

```dart
import 'package:supabase_flutter/supabase_flutter.dart';
import '../models/profile_photo.dart';

class ProfilePhotoService {
  final supabase = Supabase.instance.client;

  Future<List<ProfilePhoto>> getMyPhotos() async {
    final myId = supabase.auth.currentUser!.id;
    final response = await supabase
        .from('profile_photos')
        .select()
        .eq('user_id', myId)
        .order('display_order');

    return (response as List).map((p) => ProfilePhoto.fromJson(p)).toList();
  }

  // photoUrl must already point to an uploaded image — no storage bucket
  // exists yet in these migrations, so the upload pipeline is out of scope here.
  Future<ProfilePhoto> addPhoto({
    required String photoUrl,
    required int displayOrder, // 1-4, must be a free slot
    bool isPrimary = false,
  }) async {
    final myId = supabase.auth.currentUser!.id;
    final response = await supabase.from('profile_photos').insert({
      'user_id': myId,
      'photo_url': photoUrl,
      'display_order': displayOrder,
      'is_primary': isPrimary,
    }).select().single();

    return ProfilePhoto.fromJson(response);
  }

  Future<void> setPrimary(String photoId) async {
    await supabase.rpc('set_primary_profile_photo', params: {'photo_id': photoId});
  }

  Future<void> deletePhoto(String photoId) async {
    await supabase.from('profile_photos').delete().eq('id', photoId);
  }
}
```

Error handling to wire up:

```dart
try {
  await profilePhotoService.addPhoto(photoUrl: url, displayOrder: 3);
} on PostgrestException catch (e) {
  if (e.code == '23505') {
    // slot already taken, or (in a race) already-primary conflict — refresh and retry with a free slot
  } else if (e.code == '23514') {
    // empty photo_url or display_order out of 1-4 range
  }
}

try {
  await profilePhotoService.setPrimary(photoId);
} on PostgrestException catch (e) {
  // RPC raises plain exceptions ('Not authenticated' / 'Photo not found' /
  // 'Not the owner of this photo'), surfaced as PostgrestException with e.message
}
```

## 9. Still not built

- No storage bucket / upload pipeline — `photo_url` must come from somewhere else (matches the same gap noted for chat media in `CHAT_INTEGRATION_GUIDE.md`).
- No RPC for adding a photo or reordering `display_order` — direct table access under RLS is currently the only path.
- No auto-promotion of a new primary photo when the current primary is deleted.
- No max-photo-count-friendly "replace slot" RPC — the client must find a free `display_order` itself or handle `23505`.
